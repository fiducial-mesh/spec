# PCS Plugin Project Structure Specification

Version: 0.1
Status: Draft

This document defines the canonical directory layout, file conventions,
and discovery contracts for a PCS-governed plugin project. The structure
is substrate-agnostic — it compiles to Claude Code skills, Codex
functions, Cline rules, Cursor MDC files, or any future target.

An agent given this spec and told "set up a plugin project" should
produce a conformant structure without additional guidance. An API
given a repo URL should find everything it needs by convention.

---

## Design Constraints

1. **Repository is the entry point.** The API discovers a plugin by
   repo URL. Everything the API needs lives inside the repo at
   predictable paths. No external configuration required.

2. **Convention over configuration.** File names, directory names, and
   manifest locations are fixed by spec. Tools discover artifacts by
   walking the tree, not by reading a config that says where to look.

3. **Compiles to any target.** The canonical structure contains no
   vendor-specific artifacts. Platform-specific output (SKILL.md,
   AGENTS.md, .mdc files, config.toml entries) is generated by
   compilers into a `compiled/` directory.

4. **Scales to hundreds of plugins.** A project may contain one plugin
   or hundreds. The directory structure handles both without
   reorganization.

5. **Self-describing.** Every artifact has a manifest. Every manifest
   has the same required fields. A tool can enumerate all plugins,
   skills, runbooks, and profiles by walking the tree and reading
   manifests — no implicit knowledge required.

---

## Cross-Platform Artifact Locations

The PCS canonical structure maps to existing platform conventions:

| PCS Artifact | Claude Code | OpenAI Codex | Cline | Cursor |
|-------------|-------------|--------------|-------|--------|
| `pcs.yaml` (project manifest) | `CLAUDE.md` + `.claude/settings.json` | `AGENTS.md` + `.codex/config.toml` | `.clinerules/` directory | `.cursor/index.mdc` |
| `plugins/<name>/plugin.yaml` | `.claude/skills/<name>/SKILL.md` | Tool in `config.toml` | Rule in `.clinerules/<name>.md` | Rule in `.cursor/rules/<name>.mdc` |
| `plugins/<name>/skills/*.yaml` | `@mcp.tool()` in server.py | `@mcp.tool()` | `@mcp.tool()` | `@mcp.tool()` |
| `plugins/<name>/bodies/*.md` | SKILL.md body | AGENTS.md section | .clinerules body | .mdc body |
| `profiles/nodes/*.yaml` | N/A (implicit) | N/A | N/A | N/A |
| `profiles/agents/*.yaml` | N/A (implicit) | N/A | N/A | N/A |
| `compiled/<platform>/` | `.claude/skills/` | `.codex/` | `.clinerules/` | `.cursor/rules/` |

The `compiled/` directory is the bridge. PCS owns the canonical
structure; compilers translate to platform-native layouts.

## Canonical Directory Layout

```
<project-root>/
├── pcs.yaml                          # Project manifest (required)
│
├── plugins/                          # Plugin directory (required)
│   ├── <plugin-name>/                # One directory per plugin
│   │   ├── plugin.yaml               # Plugin manifest (required per plugin)
│   │   ├── skills/                   # Skill specs
│   │   │   ├── <skill-name>.yaml     # One file per skill
│   │   │   └── ...
│   │   ├── runbooks/                 # Runbook specs
│   │   │   ├── <runbook-name>.yaml   # One file per runbook
│   │   │   └── ...
│   │   ├── bodies/                   # Prose bodies (one .md per skill)
│   │   │   ├── <skill-name>.md       # Matches skill filename
│   │   │   └── ...
│   │   └── tests/                    # Plugin-specific tests
│   │       └── ...
│   └── <another-plugin>/
│       ├── plugin.yaml
│       ├── skills/
│       └── ...
│
├── profiles/                         # Execution profiles (optional)
│   ├── nodes/                        # Hardware capabilities
│   │   └── <node-name>.yaml
│   └── agents/                       # Authority capabilities
│       └── <agent-name>.yaml
│
├── vocabulary.yaml                   # Shared definitions (optional)
│
├── compiled/                         # Generated platform output (gitignored)
│   ├── claude/                       # Claude Code SKILL.md files
│   ├── codex/                        # Codex config entries
│   ├── cline/                        # Cline rule files
│   └── cursor/                       # Cursor MDC files
│
└── .gitignore
```

## Manifests

### pcs.yaml — Project Manifest

The root entry point. An API reads this first.

```yaml
pcs_version: "0.1"                    # PCS spec version (required)
project:
  name: my-project                    # Identifier: ^[a-z][a-z0-9_-]*$ (required)
  owner: ki7mt                        # Maintainer of record (required)
  description: ""                     # One-line description (optional)
  license: MIT                        # SPDX identifier (optional)
  repository: ""                      # Git remote URL (optional)
  default_trust_tier: community       # Default for new skills (optional)
  default_lifecycle_state: draft      # Default for new skills (optional)
```

### plugin.yaml — Plugin Manifest

One per plugin directory. Describes the plugin's concern area.

```yaml
name: solar                           # Identifier: ^[a-z][a-z0-9_-]*$ (required)
version: "0.1.0"                      # Semver (required)
owner: watson                         # Maintainer of record (required)
description: ""                       # What this plugin governs (required)
concern: propagation                  # Area of concern (optional, free-form)
tags: []                              # Discovery tags (optional)
```

### Skill Spec — skills/<name>.yaml

Defined in the PCS Skill Spec v0.1 schema. Ten fields:

```yaml
name: solar-conditions                # Must match filename
version: "0.1.0"
owner: watson
lifecycle_state: active
trust_tier: blessed
effect: pure
requires_capabilities: []
inputs: {}
outputs: {}
```

### Runbook Spec — runbooks/<name>.yaml

Defined in the PCS Runbook Spec v0.1 schema. References skills by
name and version.

### Body — bodies/<name>.md

Prose instructions for the skill. Substrate-agnostic markdown. The
compiler combines spec + body into platform-specific output.

If no body exists for a skill, the compiler generates a default from
the spec metadata.

## Naming Rules

All names use the same pattern: `^[a-z][a-z0-9_-]*$`

| Artifact | Name Source | File Name |
|----------|------------|-----------|
| Project | `pcs.yaml → project.name` | N/A (root directory) |
| Plugin | `plugin.yaml → name` | Directory name must match |
| Skill | `skills/<name>.yaml → name` | Filename must match `name` field |
| Runbook | `runbooks/<name>.yaml → name` | Filename must match `name` field |
| Body | N/A | Must match corresponding skill name |
| Node Profile | `nodes/<name>.yaml → node` | Free-form but descriptive |
| Agent Profile | `agents/<name>.yaml → agent` | Must match `agent` field |

**Enforcement:** The CLI and API validate that filenames match manifest
`name` fields. A mismatch is a validation error.

## Discovery Contract

An API or tool discovers artifacts by walking the tree. No index file
required.

```
Given: repo URL
  1. Clone repo
  2. Read pcs.yaml at root → project metadata
  3. List directories in plugins/ → plugin names
  4. For each plugin:
     a. Read plugins/<name>/plugin.yaml → plugin metadata
     b. List plugins/<name>/skills/*.yaml → skill specs
     c. List plugins/<name>/runbooks/*.yaml → runbook specs
     d. List plugins/<name>/bodies/*.md → prose bodies
     e. List plugins/<name>/tests/ → test files
  5. List profiles/nodes/*.yaml → node profiles
  6. List profiles/agents/*.yaml → agent profiles
  7. Read vocabulary.yaml if present → shared definitions
     (vocabulary is project-local in v0.1; cross-project
      vocabulary composition is not defined)
```

This is the contract the `pcs-registry` API will implement.
Everything is discoverable by convention.

## Compilation Contract

Compilers read the canonical structure and produce platform-native
output in `compiled/<platform>/`.

```
Given: canonical project
  For each platform compiler:
    1. Read all skill specs from plugins/*/skills/
    2. Read matching bodies from plugins/*/bodies/
    3. Combine spec + body into platform artifact
    4. Write to compiled/<platform>/
```

### Claude Code Compiler Output

```
compiled/claude/
├── <skill-name>/
│   └── SKILL.md              # YAML frontmatter + prose body
└── ...
```

### Codex Compiler Output (future)

```
compiled/codex/
├── AGENTS.md                  # Combined instructions
└── config.toml                # MCP server entries
```

### Cline Compiler Output (future)

```
compiled/cline/
├── <skill-name>.md            # YAML frontmatter + globs + body
└── ...
```

### Cursor Compiler Output (future)

```
compiled/cursor/
├── <skill-name>.mdc           # YAML frontmatter + globs + body
└── ...
```

## Validation Contract

A conformant project passes all of:

| Check | Rule |
|-------|------|
| `pcs.yaml` exists at root | Required |
| `pcs.yaml` parses and validates | All required fields present |
| `plugins/` directory exists | Required |
| Each plugin has `plugin.yaml` | Required per plugin |
| Plugin directory name matches `plugin.yaml → name` | Must match |
| Each skill filename matches its `name` field | Must match |
| Each skill passes Skill Spec v0.1 schema | Full validation |
| Each runbook passes Runbook Spec v0.1 schema | Full validation |
| Each runbook's referenced skills exist in the same plugin or project | Cross-reference check |
| Each body has a matching skill | Warning — must be resolved or suppressed via `.pcsignore` |
| No orphan bodies (body exists without matching skill) | Warning — must be resolved or suppressed via `.pcsignore` |
| Vocabulary parses if present | Schema validation |
| Vocabulary is project-local | v0.1 scope. Cross-project vocabulary composition not defined. |

## Scaffolding

### New Project

```bash
pcs init <name> --owner <owner>
```

Creates:
```
<name>/
├── pcs.yaml
├── plugins/
├── profiles/
│   ├── nodes/
│   └── agents/
└── .gitignore
```

### New Plugin

```bash
pcs new plugin <name> --owner <owner>
```

Creates in the current project:
```
plugins/<name>/
├── plugin.yaml
├── skills/
├── runbooks/
├── bodies/
└── tests/
```

### New Skill

```bash
pcs new skill <name> --plugin <plugin> [--effect pure|mutate_local|mutate_external]
```

Creates:
```
plugins/<plugin>/skills/<name>.yaml    # Spec with defaults from pcs.yaml
plugins/<plugin>/bodies/<name>.md      # Stub body
```

### New Runbook

```bash
pcs new runbook <name> --plugin <plugin>
```

Creates:
```
plugins/<plugin>/runbooks/<name>.yaml  # Stub with one skill step
```

## Minimum Viable Project

```
my-project/
├── pcs.yaml
└── plugins/
    └── my-plugin/
        ├── plugin.yaml
        └── skills/
            └── my-skill.yaml
```

Three files. One plugin. One skill. Everything else is added as needed.

---

## Adopting an Existing Project

```bash
cd existing-project
pcs adopt --owner <owner>
```

Creates the PCS structure alongside existing files:
- Writes `pcs.yaml` at root
- Creates `plugins/` directory
- Does NOT move or modify existing files
- Does NOT overwrite existing `pcs.yaml`

The operator then creates plugins and moves/references existing skills
into the PCS structure incrementally.

## Relationship to PCS Components

| Component | What It Does With This Structure |
|-----------|----------------------------------|
| `pcs-spec` | Defines the structure (this document) and validates it |
| `pcs-control-plane` | Validates candidates against the spec: `(spec, candidate) → verdict` |
| `pcs-registry` | Indexes plugins from `plugins/` by walking the tree per the discovery contract |
| Compilers | Read from `plugins/*/skills/` + `bodies/` → write to `compiled/<platform>/` |

The directory layout is the interface between all three PCS components.
